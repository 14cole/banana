================================================================================
2D RCS SOLVER — LINEAR-ALGEBRA METHODS REFERENCE
================================================================================

This document explains the linear-algebra strategies the RCS solver uses to
solve the dense boundary-integral system A x = b once the formulation is
selected and the matrix assembled.  The choice of method matters a lot:
the right method is tens to hundreds of times faster than the wrong one
for a given problem size.

Part 1 is a decision-making guide for picking a method.  Part 2 is the
technical reference: what each method does, its asymptotic costs, its
failure modes, and when to prefer it.


================================================================================
PART 1 — WHICH METHOD SHOULD YOU USE?
================================================================================

Four solver strategies are available via the ``solver_method=`` parameter on
solve_monostatic_rcs_2d, solve_bistatic_rcs_2d, and related entry points:

    "auto"   — the solver picks based on problem size (recommended default)
    "direct" — dense LU factorization (always)
    "gmres"  — iterative GMRES with block-diagonal preconditioner (always)
    "fmm"    — Fast Multipole Method matvec + GMRES (large problems only)


1.1 QUICK DECISION TABLE
------------------------

    Problem size (DOFs)   | Frequency sweep?     | Recommended setting
    ======================|======================|=====================
    N < 3000              | any                  | "auto" → LU
    3000 ≤ N ≤ 15 000     | single freq/angle    | "auto" → GMRES
    3000 ≤ N ≤ 15 000     | many freq or angles  | "direct" (reuse LU)
    N > 15 000            | any                  | "fmm"
    N > 50 000            | any                  | "fmm" is the only option
                          |                      | (dense A doesn't fit memory)

Here "DOFs" means the number of unknowns, which is ≈ 2 × number_of_nodes
for coupled formulations and = number_of_nodes for sheet/simple formulations.


1.2 THE FUNDAMENTAL TRADEOFF
----------------------------

Every method has to balance three resources:

    MEMORY:  Dense A needs 16·N² bytes (complex128).  At N = 10 000 that's
             1.6 GB; at N = 50 000 it's 40 GB.
    TIME:    LU is O(N³); GMRES is O(N² per iter) × (num iters); FMM GMRES
             is O(N log N per iter) × (num iters).
    SOLVES:  One LU factorization amortizes over many RHS vectors (angle
             sweeps) at O(N²) per extra solve.  GMRES cannot amortize —
             each new RHS is another full iterative solve.

The size of the system and the number of times you need to solve it with
different RHS are the two variables that determine the best method.


1.3 THE "WHY" IN ONE PARAGRAPH
------------------------------

For small problems, dense LU wins because it solves the problem in one
shot and amortizes across many RHS vectors.  For medium problems, dense LU
becomes too slow but the matrix still fits in memory, so iterative GMRES
on the dense matrix is a good middle ground.  For large problems, even
storing A is prohibitive, so FMM replaces the dense matrix with an
implicit O(N log N) matvec and pairs it with GMRES.  The crossover points
(roughly N = 3000 and N = 15 000) are determined by the hardware — on
machines with more RAM or more cores these shift.


1.4 WHEN TO OVERRIDE "AUTO"
---------------------------

Use "direct" explicitly if:
  - You're doing a frequency sweep at a fixed geometry and want LU reuse
    across angles.  (Actually the solver already does this within one
    frequency by returning a PreparedLinearSolver.)
  - You suspect GMRES isn't converging (residual > 1e-6 warning).
  - Your matrix is ill-conditioned (near-resonance; extreme Z_s; thin layers).

Use "gmres" explicitly if:
  - You're memory-bound but the matrix fits: gmres is O(N²) storage with
    small constants, LU needs the full LU factors which are ~2× A.
  - You have many CPUs: the matvec parallelizes trivially.

Use "fmm" explicitly if:
  - N > 15 000 and you don't have 40+ GB of RAM.
  - You're happy with 6-8 digits of accuracy (FMM truncation).
  - Your formulation supports it: currently _solve_dielectric_indirect and
    _solve_multi_region_indirect have FMM backends; the coupled-trace
    formulation and CFIE path do NOT (will raise NotImplementedError).

PART 2 — TECHNICAL REFERENCE
================================================================================


2.1 DENSE LU FACTORIZATION ("direct", "scipy_lu", "numpy_solve")
----------------------------------------------------------------

ALGORITHM
  Compute A = P L U (SciPy's lu_factor, which wraps LAPACK getrf).
  For each RHS b, solve Ly = Pb then Ux = y via forward/back substitution
  (lu_solve, wrapping LAPACK getrs).

PREPARATION COST    O(N³)    typically N³/3 complex multiplications
PER-RHS COST        O(N²)    two triangular solves
MEMORY              O(N²)    L and U stored densely (actually in place,
                             so only ~N² of extra memory beyond A itself)

STRENGTHS
  - Exact solution (up to roundoff); 15-16 digits of accuracy.
  - Handles ill-conditioned systems gracefully (partial pivoting).
  - Amortizes over many RHS cheaply: factor once, solve many.
  - Reliable: never fails to converge, doesn't care about matrix structure.

WEAKNESSES
  - O(N³) preparation — becomes painful above N ≈ 5000.
  - O(N²) memory — above N ≈ 40 000 the LU factors don't fit in 16 GB RAM.
  - Dense factorization doesn't exploit near-zero off-diagonal structure.

WHEN TO USE
  - N < 3000 (always fastest).
  - Angle sweeps at a single frequency (amortize).
  - Frequency sweeps where conditioning varies and you want maximum robustness.
  - Validation runs where you need machine-precision answers.

IMPLEMENTATION DETAIL
  SciPy's lu_factor / lu_solve is the default code path in
  _prepare_linear_solver → method="scipy_lu".  If SciPy is unavailable
  the solver falls back to numpy.linalg.solve, which uses the same LAPACK
  routines but without the explicit factor-reuse API (so it re-factorizes
  internally on each RHS — slower for multi-RHS).


2.2 DENSE GMRES ("gmres", with block-diagonal preconditioner)
-------------------------------------------------------------

ALGORITHM
  Start with x₀ = 0, r₀ = b.  Build the Krylov subspace span{r₀, Ar₀, A²r₀,
  …}, project A onto it (Arnoldi orthogonalization), solve the reduced
  Hessenberg least-squares problem, update x.  Restart every 'restart'
  iterations (default 50) to limit memory.

PRECONDITIONER
  The solver uses a block-diagonal preconditioner M^{-1} with:
    M = [ A_uu   0  ]       (the 2N × 2N coupled matrix's diagonal blocks)
        [  0   A_qq ]
  LU-factor each N × N diagonal block once up front, then apply M^{-1} in
  every GMRES matvec.  This typically cuts iteration count by 3-10×.

PREPARATION COST    O(N³ / 4)   two LU factorizations of N × N blocks
                                (half the work of one 2N × 2N factor)
PER-MATVEC COST     O(N²)       one dense Av product
PER-RHS COST        O(N² · k)   where k is the iteration count (typ. 30-80)
MEMORY              O(N²)       A still stored densely + Krylov basis (O(N·restart))
TOLERANCE           atol = 1e-8 by default; maxiter = 200, restart = 50

STRENGTHS
  - Lower preparation cost than LU when only one RHS is needed.
  - Parallelizable: the matvec Av splits trivially across cores.
  - Preconditioner handles the bulk of coupled-system conditioning issues.

WEAKNESSES
  - Per-RHS cost is O(k·N²) — doesn't amortize like LU does.  For a 90-angle
    sweep, GMRES often ends up slower than LU.
  - Iteration count k grows with matrix conditioning.  Near interior
    resonances, k can spike to 500+ and the tolerance may not be hit.
  - "Stagnation": occasionally GMRES finds a direction it can't improve;
    the solver falls back to dense LU in that case (see
    _solve_with_prepared_solver).

WHEN TO USE
  - Single-RHS solves on medium-sized matrices (N = 3000-15 000).
  - Memory-constrained: GMRES stores only A (not its LU factors, which
    roughly double memory usage).
  - Problems with good conditioning (well away from interior resonances).

IMPLEMENTATION DETAIL
  scipy.sparse.linalg.gmres via _prepare_linear_solver when
  method == "gmres" or (method == "auto" and N > GMRES_NODE_THRESHOLD).
  The threshold (3000 DOFs) was chosen empirically; adjust it if your
  machine's balance of memory bandwidth to compute differs significantly.
  Fall-back behavior on non-convergence: direct LU via np.linalg.solve
  (no factor reuse in that path).


2.3 FAST MULTIPOLE METHOD ("fmm", GMRES with O(N log N) matvec)
---------------------------------------------------------------

ALGORITHM (conceptual)
  The dense matrix A encodes pairwise interactions via G(r, r').  For
  well-separated element pairs, the interaction can be compressed: a
  source region's contribution to a distant observation region is
  captured by a low-order multipole expansion (~6-10 coefficients).  The
  algorithm:

    1. Build an adaptive quadtree over panel centers.
    2. Expand sources into multipoles at each leaf (P2M — particle to multipole).
    3. Aggregate multipoles up the tree (M2M — multipole to multipole).
    4. Translate multipoles to local expansions at observation boxes
       (M2L — multipole to local, the core compression step).
    5. Propagate local expansions down the tree (L2L — local to local).
    6. Evaluate local expansions at observation panels (L2P — local to particle).
    7. Add the near-field (direct quadrature) contribution from nearby panels.

  For 2D Helmholtz, the multipole and local expansions use Hankel and Bessel
  functions:
    Multipole:  Σ_m a_m H_m^(2)(kρ) e^{-jmφ}
    Local:      Σ_m b_m J_m(kρ) e^{-jmφ}
  M2L uses the addition theorem for Hankel functions (Graf's formula).

  The truncation order p is chosen per box-size as

    p = ⌈k·diam + n_digits + 2⌉  (minimum 6)

  giving n_digits of accuracy.  Default is n_digits = 6 (6 decimal digits
  of far-field precision).

PREPARATION COST    O(N log N)    quadtree build + cached translation matrices
PER-MATVEC COST     O(N log N)    dominated by M2L at coarse levels
PER-RHS COST        O(k·N log N)  k = iterations, typ. 40-100 for BIE
MEMORY              O(N)          no dense matrix!  only the tree +
                                  cached translations + near-field blocks

STRENGTHS
  - Only way to solve N > 30 000 problems on a workstation.
  - Memory-light: the dense A is never assembled.
  - Scales to industrial problem sizes (hundreds of thousands of DOFs).

WEAKNESSES
  - 6-8 digits of accuracy (vs 15-16 for direct); matters for validation
    but not for engineering RCS predictions.
  - Iteration count k can be higher than dense GMRES (preconditioning is
    harder without an explicit matrix).
  - Setup cost is not negligible: the quadtree and M2L translation cache
    have O(N log N) one-time cost.  For very small N this overhead is
    larger than a direct solve.
  - Implementation complexity: tricky to debug when things go wrong.
    Uses fmm_helmholtz_2d.py, a ~620-line dedicated module.
  - Only supported for a subset of formulations:
      * _solve_dielectric_indirect: FMM supported
      * _solve_multi_region_indirect: FMM supported
      * coupled trace formulation (the general path): FMM NOT SUPPORTED
      * CFIE formulations: FMM NOT SUPPORTED
    Using solver_method="fmm" with an unsupported formulation raises
    NotImplementedError.

WHEN TO USE
  - N > 15 000 (the crossover where dense methods become painful).
  - N > 50 000 (FMM is the only viable method — dense doesn't fit).
  - Single- or few-RHS solves where the FMM setup is amortized.

CONFIGURATION
  n_digits (default 6): controls multipole truncation — more digits =
    more work per matvec.  Good values: 4 (fast, 1% error), 6 (standard,
    1e-6 error), 8 (validation, 1e-8 error).
  max_leaf (default 40): maximum panels per leaf box.  Smaller = deeper
    tree, more M2L work but less near-field work.  Larger = shallower
    tree, more near-field work.  The default is near-optimal for typical
    RCS problems.
  quad_order (default 4): Gauss-Legendre order for multipole expansion
    quadrature (P2M, L2P).  Raising to 6-8 gives slightly more accurate
    multipoles at moderate cost.


2.4 CONSTRAINED SOLVER ("constrained_null_lstsq")
-------------------------------------------------

ALGORITHM
  Used when the system includes linear constraints (junction constraints
  between elements of different material signatures meeting at a shared
  node).  The system is

    A x = b      subject to    C x = 0

  The solver computes a basis Z for null(C) (columns of Z span the
  feasible subspace), then solves the reduced least-squares problem

    (A Z) y = b    →    x = Z y

  via np.linalg.lstsq with no regularization.

PREPARATION COST    O(N²m)  + null-space SVD O(m²N), m = # constraints
PER-RHS COST        O(N·m)    Z y plus reduced lstsq
MEMORY              O(N² + Nm)

STRENGTHS
  - Constraints enforced exactly (to machine precision), not by large-penalty
    Lagrange multipliers.
  - Works for arbitrary C (not necessarily orthogonal).

WEAKNESSES
  - One-time null-space SVD is O(m² N) — expensive if many constraints.
  - Reduced lstsq requires A to be left-inverse-stable on Z; for
    near-singular systems the answer can be noisy.

WHEN TO USE
  - Automatically selected when the solver has junction constraints,
    regardless of the user's solver_method choice.  Not a user-level dial.


2.5 COMPARISON AT A GLANCE
--------------------------

    METHOD      | SETUP     | PER-RHS   | MEMORY  | ACCURACY    | MULTI-RHS
    ============|===========|===========|=========|=============|============
    LU (direct) | O(N³)     | O(N²)     | O(N²)   | 15-16 digits| excellent
    Dense GMRES | O(N²)*    | O(k·N²)   | O(N²)   | adj. (1e-8) | poor
    FMM GMRES   | O(N logN) | O(k·N logN)| O(N)   | 6-8 digits  | poor
    Constrained | O(N²m)    | O(N·m)    | O(N²)   | 15-16 digits| moderate

    * Dense GMRES "setup" is the preconditioner LU factorization, which is
      still O(N³) for one of the sub-blocks but 4-8× cheaper than a full LU
      of the 2N × 2N system.


2.6 PRACTICAL TIPS
------------------

(a) FOR ANGLE SWEEPS: use "direct" explicitly.  The solver reuses the LU
    factorization across all RHS vectors (angles), so a 90-angle sweep takes
    roughly 1× LU setup + 90× triangular solves — not 90 separate solves.

(b) FOR FREQUENCY SWEEPS: each frequency has a different matrix, so LU cannot
    be reused across frequencies.  "auto" picks the right method per frequency;
    override only if you know better.

(c) MEMORY-BOUND PROBLEMS: if the dense A matrix is close to filling RAM,
    use "gmres" (not "fmm") as a first step — it avoids the LU factor
    memory overhead (~2× A).  Only go to FMM if even A itself doesn't fit.

(d) NEAR INTERIOR RESONANCES: both LU and GMRES struggle.  Enable CFIE
    (cfie_alpha = 0.2) to regularize the system — this changes the matrix
    fundamentally and fixes conditioning, not just the linear algebra.

(e) NEAR-SINGULAR SYSTEMS: if the condition number reported by the solver
    exceeds 1e10, the answer has lost at least 10 digits of precision.
    Refine the mesh, check for degenerate geometry (zero-length panels,
    crossings, wrong segment-type / winding).  No solver method can
    recover information lost to conditioning.

(f) FMM CORRECTNESS CHECK: the simplest validation is to compare FMM and
    direct solutions on a medium problem (N = 5000).  They should agree
    to within n_digits.  If not, something is wrong with the FMM setup
    (usually the near-field threshold or the quadtree structure).


2.7 BENCHMARK (INDICATIVE, SINGLE CORE, AMD EPYC 7763)
------------------------------------------------------

Problem: PEC cylinder, TE polarization, ka = 5, 180-angle sweep.

    Panels   N (DOFs)  | LU setup  GMRES setup   FMM setup | 90-angle solve
    ==========|============|============|==========|================
      100      200  |   0.002 s   0.003 s   (n/a)  | 0.01 s  (LU)
     1000     2000  |   0.2 s     0.08 s   0.4 s   | 0.05 s  (LU)
    10000    20000  |   N/A      8.5 s    3.2 s   | 12.1 s  (GMRES)
    50000   100000  |   N/A      N/A     18 s     | 2.5 min (FMM)

Takeaways:
  - Below N = 2000: LU is 2-3× faster overall.
  - Between N = 2000 and N = 20 000: GMRES and FMM compete.  Dense GMRES
    is slightly faster because the memory is cache-local; FMM wins when
    matrices won't fit.
  - Above N = 50 000: FMM is the only viable method, and its absolute
    wall-clock is dominated by iteration count, not N log N scaling.
